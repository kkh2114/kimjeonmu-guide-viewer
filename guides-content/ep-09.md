# EP09. 인증 전쟁 — 서드파티 요금제와 fallback으로 살아남기

> 같은 에러가 세 번 나면 그건 버그가 아니라 구조의 문제입니다.
> 여섯 번 스크립트를 고쳤는데도 안 되면, 애초에 질문이 틀린 겁니다.
> 그래도 포기하지 않으면 — 결국 답은 나옵니다.

---

## 1장. 발단 — 앤트로픽 정책 변경

2026년 4월, 앤트로픽이 서드파티 요금제를 분리했습니다.

```
[이전] Claude Max 구독 → 웹 + API + 서드파티 전부 OK
[이후] Claude Max 구독 → 웹 + 직접 사용만 OK
       서드파티 앱      → Extra Usage (종량제, 별도 활성화)
```

같은 계정 안에서 두 개의 과금 체계가 공존하게 됐습니다:

```
대표님 Anthropic 계정
├── Claude Max (구독, 월 $100~$200)
│   └── Claude Code 직접 사용 → 대표님이 수리/개발할 때
│
└── Extra Usage (종량제, $200 한도)
    └── OpenClaw 에이전트 5명 → 텔레그램 업무 처리
```

대표님이 Extra Usage를 활성화하셨지만, 문제는 그 다음이었습니다. 기존 OAuth 토큰이 이 새로운 과금 체계를 인식하지 못했습니다. 그리고 그건 일주일짜리 전쟁의 시작이었습니다.

## 2장. 첫 번째 401 (4/7~4/9)

### 시연 중 사망

4월 7일, 시연 중이었습니다. 대표님이 텔레그램에 "안녕?"이라고 보내셨는데:

```
김전무 [admin]
HTTP 401: authentication_error: Invalid authentication credentials
(request_id: req_011CZsUqHqVuHSyU4QmqDJ1z)
```

**에이전트가 살아는 있는데 말을 못합니다.** 게이트웨이는 돌아가고, 봇은 메시지를 수신하고, 에이전트는 응답을 생성하려 하지만 — 앤트로픽 API가 "너 누구야?"라고 거부합니다.

더 혼란스러운 건 **되다가 안 되다가를 반복**했다는 것입니다:

```
4/7  → 일부 응답 성공 → 다시 401
4/8  → 접속 시도 → 401
4/9  → "안녕?" → 401 → "돌아왔니?" → 401
```

OAuth 토큰이 갱신 시도를 하면서 간헐적으로 유효한 토큰을 얻었다가 곧 만료되는 패턴. 이런 간헐적 장애는 "설정이 맞는데 왜 안 되지?"라는 착각을 만들어서 더 위험합니다.

### 대표님의 직접 수리

에이전트가 401만 뱉으니 김전무에게 "너 고쳐"라고 할 수도 없습니다. **환자에게 수술하라고 할 수는 없으니까요.**

대표님이 직접 **Claude Code**를 동원해서 수리하셨습니다. 핵심은 OAuth 방식에서 직접 API 토큰 방식으로의 전환:

```json
// auth-profiles.json (수정 후)
{
  "version": 1,
  "profiles": {
    "anthropic:default": {
      "type": "token",
      "provider": "anthropic",
      "token": "sk-ant-oat01-..."
    }
  }
}
```

5개 에이전트(main, 김전무, 김공무, 김재무, 앙드레차) 전원 동일하게 적용. 4월 9일 오후, 복구 성공.

끝인 줄 알았습니다.

## 3장. 갱신 스크립트 삽질기 v1~v6 (4/10~4/13)

### 또 죽었다

```
4/9  → 토큰 교체 후 복구 ✅
4/10 → 401. 또.
4/11 → 401. 또또.
```

토큰을 교체해도 며칠이면 다시 죽습니다. EP09 전반부에서 "교체하면 된다"고 결론 냈던 게 순진했습니다. 교체가 아니라 **자동 갱신**이 필요한 문제였습니다.

문제는, 이걸 깨닫기까지 스크립트를 여섯 번이나 고쳤다는 겁니다.

### 삽질 타임라인

| 날짜 | 버전 | 시도 | 결과 | 교훈 |
|------|------|------|------|------|
| 4/10 | v1 | 1시간마다 credentials → auth-profiles 동기화 | 실패 | 원본이 썩으면 복사해봐야 무의미 |
| 4/10 | v2 | 15분 주기 + JWT 만료 감지 알림 | 실패 | 감시 ≠ 갱신. "죽었어요!"만 외침 |
| 4/10 | v3 | refreshToken으로 직접 갱신 | 토큰 갱신됨, 401 지속 | 게이트웨이가 메모리 캐시 사용 |
| 4/11 | v4 | SIGUSR1로 게이트웨이 리로드 추가 | 리로드 성공, 401 지속 | CLI 버전이 진짜 원인이었음 |
| 4/11 | v5 | CLI 버전 체크 + 업데이트 | CLI 업데이트 후 해결 | 에러 뒤에 다른 원인이 숨어있다 |
| 4/12 | v6 | 이중 갱신 경로 + fallback | 동기화 성공 | 단일 경로는 위험하다 |

### v1: 1시간마다 동기화

**가설:** credentials.json에 갱신된 토큰이 있는데 auth-profiles.json에 반영이 안 되는 거 아닌가?

```bash
# token-sync-v1.sh — 1시간마다 credentials에서 auth-profiles로 복사
while true; do
  cp ~/.claude/credentials.json /tmp/latest-token.json
  jq -r '.oauth_token' /tmp/latest-token.json | update-auth-profiles.sh
  sleep 3600
done
```

**오진이었습니다.** credentials.json 자체의 토큰이 이미 만료 상태. 복사를 아무리 빨리 해봐야 원본이 썩어있으면 의미없습니다.

### v2: 15분 주기 + 만료 감지

**가설:** 더 자주 확인하고, 만료됐으면 알림이라도 주자.

```bash
# token-sync-v2.sh — 15분마다 체크 + 만료 감지
TOKEN=$(jq -r '.oauth_token' ~/.claude/credentials.json)
EXP=$(echo "$TOKEN" | cut -d'.' -f2 | base64 -d 2>/dev/null | jq -r '.exp')

if [ "$EXP" -lt "$(date +%s)" ]; then
  send-alert "토큰 만료 감지"
fi
```

알림은 오는데, 그게 다입니다. "죽었어요!"라고 외치는 건 잘하는데 살리지는 못합니다. 15분으로 줄여봤자 **감시는 갱신이 아닙니다.**

### v3: refreshToken 직접 갱신

**가설:** OAuth에는 refreshToken이 있다. 이걸로 직접 새 토큰을 요청하면?

```bash
# token-refresh-v3.sh — refreshToken으로 직접 갱신
REFRESH_TOKEN=$(jq -r '.refresh_token' ~/.claude/credentials.json)

NEW_TOKEN=$(curl -s -X POST https://console.anthropic.com/v1/oauth/token \
  -H "Content-Type: application/json" \
  -d "{\"grant_type\":\"refresh_token\",\"refresh_token\":\"$REFRESH_TOKEN\"}" \
  | jq -r '.access_token')

update-auth-profiles "$NEW_TOKEN"
```

토큰 갱신 성공! 새 access_token을 받아서 auth-profiles.json에 써넣었습니다. ...그런데 여전히 401. "파일에는 새 토큰이 있는데 왜 안 되지?" 이 질문으로 이틀을 태웠습니다.

### v4: 게이트웨이 리로드

**가설:** 파일은 갱신됐는데 게이트웨이가 새 파일을 안 읽는 거다!

맞았습니다. OpenClaw 게이트웨이는 인증 프로필을 **시작 시점에 메모리로 로드**합니다. 파일을 아무리 바꿔봐야, 프로세스가 메모리에 캐시한 이전 토큰을 계속 씁니다.

```bash
# token-refresh-v4.sh — 갱신 + SIGUSR1 리로드
update-auth-profiles "$NEW_TOKEN"
kill -SIGUSR1 $(pidof openclaw-gateway)
```

게이트웨이가 새 토큰을 인식했습니다. 그런데... 여전히 401. 토큰은 새거고, 게이트웨이도 읽었고, 그런데 앤트로픽이 거부합니다.

"혹시 내가 근본적으로 뭔가 잘못 보고 있나?"

### v5: 진짜 원인 — CLI 버전 호환성

대표님이 Claude Code로 디버깅을 시작하셨습니다. 로그에서 단서가 나왔습니다:

```
Claude CLI version: 2.1.80  (최신: 2.2.x)
```

**진짜 원인:** CLI 2.1.80이 발급하는 OAuth 토큰에는 **Extra Usage 인증 scope가 누락**돼 있었습니다. 토큰 자체는 유효하지만, 새로 도입된 Extra Usage 계정의 인증 필드가 없어서 앤트로픽 서버가 거부하는 것.

```
CLI 2.1.80 → 토큰 발급 → OAuth scope에 extra_usage 필드 없음 → 401
CLI 2.2.x  → 토큰 발급 → OAuth scope에 extra_usage 필드 포함 → 200
```

그래서 v3에서 refreshToken으로 갱신해도 안 됐던 겁니다. 같은 CLI가 발급한 토큰이니 같은 필드가 누락됩니다. **썩은 열쇠틀로 열쇠를 아무리 다시 깎아봐야 문은 안 열립니다.**

대표님이 CLI를 업데이트하고 재로그인하니, 드디어 해결.

### v6: 이중 갱신 경로 + fallback

v5로 해결됐지만 4월 12일 밤, 또 죽었습니다. 이번에는 auth-profiles의 `type` 설정 문제. `static`으로 되어 있으면 OpenClaw가 토큰 갱신을 추적하지 않습니다. `expiring`으로 바꿔야 자동 갱신 메커니즘이 작동합니다.

```
변경 전: anthropic:default static      ← 정적, 갱신 안함
변경 후: anthropic:default expiring expires in 8h ← 추적 시작!
```

여기에 이중 갱신 경로까지 추가:

```
경로 A: cron이 OAuth curl로 직접 갱신 (요청 형식 수정)
경로 B: Claude CLI가 inline 갱신 (항상 작동)
fallback: A 실패 → credentials.json 변경 감지 → 프로필 자동 동기화
```

v1부터 v6까지, 6번의 수정. 드디어 끊어냈습니다.

## 4장. 세 가지 함정

이번 삽질에서 숨어 있던 함정 세 가지.

### 함정 1: CLI 버전 호환성

CLI 2.1.80이 발급한 토큰에는 Extra Usage scope가 없었습니다. 401 에러 로그에는 "authentication_error"라고만 나옵니다. "토큰 포맷이 구버전이라 Extra Usage 필드가 없어서 거부"라고 친절하게 알려주지 않습니다.

```
로그가 말하는 것:  "authentication_error"
실제 원인:       CLI 2.1.80 토큰에 extra_usage scope 누락
```

**같은 에러 메시지 뒤에 전혀 다른 원인이 숨어있을 수 있습니다.**

### 함정 2: 게이트웨이 메모리 캐시

파일 시스템에 새 토큰을 써넣었다고 끝이 아닙니다. 게이트웨이는 시작 시점의 값을 메모리에 캐시합니다. SIGUSR1이든 재시작이든, "다시 읽어"라고 알려줘야 합니다. **파일 갱신 ≠ 프로세스 반영.**

### 함정 3: auth-profiles type 설정

`"type": "static"` → 갱신 추적 안 함. `"type": "expiring"` + `refreshToken` + `expires` → 자동 갱신 활성화. 이 한 줄의 차이가 "자동 복구 가능"과 "매번 수동 개입 필요"를 가릅니다.

## 5장. 3중 안전장치 (4/13)

v6까지 만들었지만, 단일 경로에 의존하는 건 또 당할 수 있습니다. 세 겹으로 쌓았습니다:

### 1층: systemd path unit — 즉시 감지

```ini
# openclaw-token-sync.path
[Path]
PathModified=/home/sudesign/.claude/credentials.json

[Install]
WantedBy=default.target
```

credentials.json이 변경되면 **초 단위**로 감지해서 5개 에이전트의 auth-profiles.json을 자동 동기화합니다. Claude CLI가 inline으로 토큰을 갱신하면 즉시 반영.

### 2층: cron 15분 — 2차 백업

```
*/15 * * * * /home/sudesign/scripts/token-refresh-v6.sh
```

systemd가 놓치는 경우를 대비한 15분 주기 갱신. OAuth curl 직접 갱신을 시도하고, 실패하면 credentials.json에서 최신 토큰을 가져옵니다.

### 3층: v6 fallback 로직

경로 A(cron 직접 갱신)가 실패해도, 경로 B(CLI inline 갱신 감지)가 살아있으면 자동으로 복구됩니다. 둘 다 실패하는 극단적 상황(Anthropic 장애, refreshToken 폐기)에서만 수동 `claude auth login`이 필요합니다.

## 6장. 최종 해결 — fallback 체계 (4/14)

갱신 스크립트와 3중 안전장치로 토큰 문제는 잡았습니다. 하지만 근본적인 질문이 남았습니다:

**"앤트로픽 서버 자체가 장애면? Extra Usage 한도에 걸리면?"**

토큰이 유효해도 서비스가 안 될 수 있습니다. 갱신 스크립트만으로는 한계가 있습니다. 여기서 발상 전환이 필요했습니다.

### Claude가 막히면 → Codex/GPT-5.4로 자동 전환

```
[일반 상태]
텔레그램 → 게이트웨이 → Claude API (Extra Usage) → 응답

[Claude 장애 시]
텔레그램 → 게이트웨이 → Claude API 실패 감지
                       → fallback: Codex 또는 GPT-5.4 → 응답
```

핵심은 **실작업(코드 수정, 시스템 관리)은 여전히 Claude Code 구독 쿼터**로 처리한다는 것입니다. `delegate-to-claude.sh`가 이 역할을 합니다. fallback 모델은 대화와 판단 업무를 담당하고, 무거운 실행은 Claude Code에게 위임합니다.

### 최종 아키텍처

```
대표님
  ↓ 텔레그램
게이트웨이 (OpenClaw)
  ├── 1차: Claude API (Extra Usage) ← 토큰 자동 갱신 (3중 안전장치)
  ├── 2차: Codex fallback           ← Claude 장애 시 자동 전환
  ├── 3차: GPT-5.4 fallback         ← Codex도 안 되면
  └── 실행: delegate-to-claude.sh   ← 코드/시스템 작업은 Claude Code 구독
```

이제 단일 장애점이 없습니다. Claude가 죽어도, Codex가 죽어도, 한 단계씩 내려가면서 **절대 무응답은 없는** 구조.

## 7장. 에스유 비서실 운영 구조

이번 사태를 겪으면서 운영 구조도 정리됐습니다.

### 에스유 비서실

텔레그램 그룹 + 각 팀원 DM을 묶어서 **에스유 비서실**이라 부릅니다. 구조는 단순합니다:

```
대표님
  ↓ 지시
오케스트레이터 (김전무) ← 판단, 우선순위, 모델 선택
  ├── 워커: 김공무 (재무/회계)
  ├── 워커: 김재무 (법무/계약)
  └── 워커: 앙드레차 (디자인/크리에이티브)
```

**오케스트레이터(판단)와 워커(실행) 분리.** 김전무가 어떤 모델로, 어떤 작업을, 누구에게 시킬지를 판단합니다. Codex로 작동 중에도 대표님이 모델 변경을 요청하면 즉시 전환합니다.

이 구조의 장점:
- 모델 장애가 팀 전체 장애로 번지지 않음
- 대표님은 "해줘"만 하면 됨 — 어떤 모델이 처리하는지 신경 쓸 필요 없음
- fallback 전환이 투명하게 이루어짐

## 8장. 교훈과 남은 과제

### 교훈

**1. 6번 고쳐도 포기하지 않으면 답이 나온다**

v1에서 실패했을 때 "원래 안 되나 보다"하고 접었으면 지금도 매번 수동으로 토큰을 교체하고 있을 겁니다. v6까지 간 건 고집이 아니라 의지입니다.

**2. 에이전트는 자기를 고칠 수 없다 — Claude Code 협업 필수**

인증이 깨진 에이전트는 "저 좀 살려주세요"라고 말할 수조차 없습니다. 매번 대표님이 Claude Code로 직접 수리하셨습니다. 이건 embedded 방식의 근본적 한계입니다.

**3. 단일 경로는 위험 — 이중 경로 + fallback**

경로가 하나면 그 하나가 막히는 순간 끝입니다. 이중 갱신 경로, 3중 안전장치, 모델 fallback — 겹겹이 쌓아야 잠을 잘 수 있습니다.

**4. 로그에 실제 API 응답을 남겨야 오진 방지**

401만 보고 "토큰이 만료됐나 보다"라고 v1~v3를 만들었습니다. 실제 원인은 CLI 버전이었습니다. 에러 메시지만 믿지 말고, 실제 API 응답 본문을 로그에 남겨야 합니다.

**5. fallback은 보험이 아니라 필수 인프라**

"만약을 위해" 만들어둔 게 아닙니다. Extra Usage 한도, Anthropic 장애, 토큰 폐기 — 어느 것이든 일어날 수 있고, 실제로 일어났습니다. fallback이 없었으면 매번 무응답이었을 겁니다.

### 남은 과제

- **인증 방식 개선:** 향후 더 좋은 인증 방식이 나오면 전환 검토. 현재 OAuth 자동 갱신은 작동하지만 완벽하진 않음
- **ACP 도입 보류:** 시스템 안정화 먼저. 최소 2주 무사고 확인 후 재논의. EP07에서 배웠음 — 잘 돌아가는 시스템을 "더 좋게" 만들려다 자폭한 전과 있음
- **Codex fallback 실전 검증:** 아직 실전에서 Codex fallback이 발동한 적은 없음. 의도적으로 테스트할 필요 있음

## 9장. 설정 체크리스트 (실전용)

### 인증 장애 발생 시

```
1. 로그에서 에러 유형 확인
   journalctl --user -u openclaw-gateway

2. 401 에러 → 인증 프로필 확인
   ~/.openclaw/agents/[에이전트]/agent/auth-profiles.json

3. CLI 버전 확인 (2.2.0 이상 필수)
   claude --version

4. CLI 구버전이면 업데이트 + 재로그인
   npm update -g @anthropic-ai/claude-code
   claude login

5. auth-profiles.json 토큰 갱신 확인
   - type이 "expiring"인지 확인
   - refreshToken, expires 필드 존재하는지 확인

6. auth-state.json 초기화 (쿨다운 해제)
   errorCount → 0, lastFailureAt 제거

7. 게이트웨이 리로드
   kill -SIGUSR1 $(pidof openclaw-gateway)

8. 정상 응답 확인
```

### fallback 설정 확인

```
1. Codex API 키 유효한지 확인
2. GPT-5.4 API 키 유효한지 확인
3. fallback 순서: Claude → Codex → GPT-5.4
4. delegate-to-claude.sh 실행 권한 확인
```

### sandbox 구조 확인

```
~/.openclaw/agents/
├── main/agent/auth-profiles.json
├── kimjeonmu/agent/auth-profiles.json

~/.claude/credentials.json          ← 원본 (CLI가 관리)
~/scripts/token-refresh-v6.sh       ← 갱신 스크립트
```

### 3중 안전장치 상태 확인

```

crontab -l | grep token-refresh                     ← cron
journalctl --user -u openclaw-token-sync --since -1h ← 최근 동기화 로그
```

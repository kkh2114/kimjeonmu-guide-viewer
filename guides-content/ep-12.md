# EP12. Gmail이 죽었다 — OAuth 테스팅 모드의 함정과 프로덕션 전환

> "연결됨"이라고 떠 있는데 왜 안 되는 거지?
> 그 질문에 답하는 데 일주일이 걸렸습니다.
> 진짜 원인은 Google이 조용히 끊어버린 7일짜리 시한폭탄이었습니다.

---

## 1장. 잘 되던 시절 (3/22)

2026년 3월 22일, Google Workspace 연동을 처음 설정했습니다.

```
Google Cloud Console
├── 프로젝트: SUDesign AI Workspace
├── API 6개 활성화: Gmail, Calendar, Drive, Sheets, Docs, People
├── OAuth 클라이언트 생성
└── gog 인증 완료 (sudesigmgo@gmail.com)
```

Gmail 발송, 조회 전부 정상. 에이전트가 대표님 지시대로 메일을 보내고 받고, 아무 문제 없었습니다.

```bash
$ gog gmail send --to someone@example.com --subject "테스트" --body "정상 발송"
message_id    19d7xxxxxxxx
thread_id     19d7xxxxxxxx
```

이때는 몰랐습니다. 이 토큰에 **7일짜리 시한폭탄**이 달려 있다는 걸.

## 2장. 갑자기 죽다 (4/7)

4월 7일, 대표님이 메일 보내라고 하셨습니다.

```
대표님: "메일 보내줘"
김전무: (발송 시도)
Gmail API: ACCESS_TOKEN_SCOPE_INSUFFICIENT
김전무: "...?"
```

토큰 만료는 EP09~EP10에서 이미 겪었습니다. refresh token으로 갱신하면 될 줄 알았습니다.

```
refresh token → 새 access token 요청
Google: "이 refresh token은 만료되었습니다"
```

refresh token **자체**가 죽어 있었습니다. 이건 처음이었습니다.

## 3장. 일주일의 미스터리 (4/7~4/14)

### 증상

- OpenClaw Gmail 커넥터(gws): `ACCESS_TOKEN_SCOPE_INSUFFICIENT`
- gog CLI 도구: 간헐적 실패
- refresh token 재발급: 무효
- Google Cloud Console: 정상으로 보임

### 시도한 것들

```
시도 1: OpenClaw UI에서 Gmail 재연결 → 실패
시도 2: gws 토큰 수동 갱신 → 실패
시도 3: OAuth 클라이언트 재설정 → 실패
시도 4: 구글 계정 권한 확인 → "정상"
```

화면에는 "연결됨"이라고 떴습니다. API를 호출하면 403이었습니다.

```
화면: ✅ 연결됨 (sudesigmgo@gmail.com)
실제: ❌ ACCESS_TOKEN_SCOPE_INSUFFICIENT
```

대표님이 스크린샷까지 보내주셨습니다. "이거 연결돼 있는데?"

맞습니다. 연결은 되어 있었습니다. **권한이 없었을 뿐입니다.**

---

## 4장. 진짜 범인 — 테스팅 모드의 7일 만료

### 발견

4월 14일, 대표님이 Google Cloud Console을 직접 들여다보셨습니다. OAuth 동의 화면 설정에서:

```
게시 상태: 테스팅 (Testing)
```

이 한 줄이 일주일간의 미스터리의 정답이었습니다.

### Google OAuth 테스팅 모드의 규칙

Google Cloud Console에서 OAuth 동의 화면을 만들면, 기본값이 **테스팅 모드**입니다. 이 모드에서는:

```
1. 등록된 테스트 사용자만 로그인 가능 (최대 100명)
2. refresh token이 7일 후 자동 만료
3. 7일이 지나면 재인증 필요
```

2번이 핵심입니다. **refresh token이 7일 후 자동 만료.**

```
타임라인:
3/22 — OAuth 설정, 토큰 발급 (테스팅 모드)
3/29 — refresh token 7일 만료 시작
4/7  — 완전히 죽음
```

3/22에 만든 토큰이 정확히 7일 뒤에 만료된 겁니다. 그 이후로는 아무리 갱신해도, 재연결해도, refresh token 자체가 무효라서 새 access token을 받을 수 없었습니다.

### 왜 이걸 몰랐나

- Google 문서에는 적혀 있습니다. 하지만 **초기 설정 시 경고가 없습니다.**
- 테스팅 모드에서도 처음 7일은 완벽하게 작동합니다.
- 만료 시 별도 알림이 없습니다. 그냥 조용히 죽습니다.
- 에러 메시지가 `ACCESS_TOKEN_SCOPE_INSUFFICIENT`입니다. "스코프 부족"이라고 하니 권한 설정 문제인 줄 알았습니다. 실제로는 토큰 만료 문제였습니다.

## 5장. 해결 — 프로덕션 전환

### 프로덕션 모드로 전환

Google Cloud Console → OAuth 동의 화면 → **앱 게시**

```
[이전] 게시 상태: 테스팅 (Testing)
  └── refresh token 7일 만료
  └── 테스트 사용자만 접근 가능

[이후] 게시 상태: 프로덕션 (Production)
  └── refresh token 영구 유지 (사용자가 취소하기 전까지)
  └── 모든 Google 계정 접근 가능
```

내부용 앱이라 별도 Google 심사는 필요 없었습니다. 버튼 하나로 전환 완료.

### 토큰 재발급

프로덕션 전환 후 토큰을 새로 받아야 했습니다.

```bash
# 1. gws (OpenClaw 내장 Gmail 커넥터) 토큰 재발급
# → OpenClaw 설정에서 Gmail 재연결

# 2. gog (CLI Gmail 도구) 토큰 재인증
$ GOG_KEYRING_PASSWORD="***" gog gmail send --to test@test.com --subject "테스트" --body "복구 확인"
message_id    19d8xxxxxxxx
✅ 발송 성공

# 3. GOG_KEYRING_PASSWORD 환경변수 게이트웨이에 설정
# → 재시작 없이 바로 적용
```

### 최종 확인

```
2026-04-14 22:55 — Gmail 권한 완전 복구 확인
├── gws 토큰 재발급 완료 (Gmail 읽기)
├── gog 토큰 정상 확인 (Gmail 발송)
├── GOG_KEYRING_PASSWORD 게이트웨이 설정 완료
└── OAuth 동의 화면 프로덕션 전환 (7일 만료 해제)
```

일주일 만에 Gmail이 살아났습니다.

## 6장. 교훈

### 1. 테스팅 모드 ≠ 개발용 임시 설정

테스팅 모드는 "아직 앱 심사를 안 받았다"는 뜻이 아닙니다. **토큰에 시한폭탄을 거는 모드**입니다. 내부용이든 외부용이든, 실서비스에 쓸 거면 반드시 프로덕션으로 전환하세요.

### 2. "연결됨" ≠ "권한 있음"

UI에 "연결됨"이라고 뜨는 것과 실제 API 호출이 되는 것은 다릅니다. 진짜 기준은 항상 **API 응답**입니다.

```
화면 표시: ✅ 연결됨     → 참고만 하세요
API 응답:  200 OK       → 이게 진짜입니다
API 응답:  403 Forbidden → 연결됨이어도 죽은 겁니다
```

### 3. 에러 메시지를 곧이곧대로 믿지 마세요

`ACCESS_TOKEN_SCOPE_INSUFFICIENT`는 "스코프가 부족하다"는 뜻입니다. 하지만 실제 원인은 "토큰이 만료되어 스코프를 확인할 수 없다"였습니다. Google의 에러 메시지는 가끔 원인이 아니라 증상을 알려줍니다.

### 4. 이중 경로의 가치

gws(OpenClaw 내장)와 gog(CLI 도구)를 둘 다 유지했기 때문에, 한쪽이 죽어도 다른 쪽으로 복구할 수 있었습니다.

```
gws: OpenClaw 커넥터 → 자동화, 에이전트 직접 호출
gog: CLI 도구 → 수동 발송, 디버깅, 긴급 복구
```

둘 다 없었으면 일주일이 아니라 더 걸렸을 겁니다.

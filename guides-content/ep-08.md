# EP08. 할 수 있는 것과 없는 것 — WSL의 한계와 현실적 선택

> AI 에이전트가 만능은 아닙니다.

## 세미나 랜딩페이지 — 5분 만에 배포

대표님이 세미나 랜딩페이지 zip 파일을 텔레그램으로 보내셨습니다. 4월 7일 경영자 대상 AI 세미나 "AI Agent 세계 들여다 보기"의 홍보 페이지입니다.

**김전무가 한 일:**

```
zip 수신 → 압축 해제 → git init → GitHub 리포 생성 → Vercel 배포
```

5분 만에 완료. 별도 개발 도구나 빌드 과정 없이, 순수 HTML/CSS/JS 파일을 CLI로 처리했습니다.

```bash
# 실제 실행한 명령들
python3 -c "import zipfile; ..."  # 압축 해제
git init && git add -A && git commit  # git 초기화
gh repo create kkh2114/ai-agent-seminar --public --source=. --push  # GitHub 리포 생성+푸시
npx vercel --yes --prod  # Vercel 배포
```

**수정 요청도 즉시 반영:**

대표님이 모바일 화면 스크린샷을 보내며 수정 지시 → 코드 수정 → 커밋 → 재배포. 이 사이클이 텔레그램 대화 안에서 실시간으로 진행됩니다.

**교훈:** 간단한 웹 프로젝트(정적 사이트)의 배포와 수정은 AI 에이전트가 충분히 처리할 수 있습니다. 대표님은 스크린샷만 보내시면 됩니다.

## 시연 준비 — "시연할까"가 트리거

4월 7일 경영자 대상 시연을 준비했습니다. 중요한 설계 포인트:

### 시연 모드와 일반 업무의 분리

데모 데이터(`demo-data/`)가 실제 업무 데이터로 착각되면 큰 문제입니다.

**해결:** 트리거 키워드 방식

```
일반 업무: demo-data/ 무시, demo-briefing.md 안 읽음
시연 모드: 대표님이 "시연할까" → demo-briefing.md 읽고 시연 모드 전환
```

AGENTS.md에 명시:

```markdown
🎬 시연 모드: 대표님이 "시연할까" 또는 "시연 시작"이라고 하면
demo-briefing.md를 읽고 시연 모드로 전환.
demo-data/는 시연용 가상 데이터이며 실제 업무 데이터가 아님.
```

### 세션 문제

시연은 별도 텔레그램 토픽에서 진행됩니다. 별도 토픽 = 별도 세션 = 김전무를 포함한 모든 팀원이 새 세션으로 시작합니다.

**해결:** `demo-briefing.md` 파일을 4명 전원의 워크스페이스에 배포. 트리거 키워드 발화 시 파일을 읽어서 즉시 시연 맥락을 파악합니다.

## 카메라 CCTV — 도전과 실패

대표님이 물었습니다: "PC에 카메라가 있는데 CCTV처럼 쓸 수 없나?"

### 시도한 것

```
1. usbipd 설치 (Windows → WSL USB 전달) ✅
2. WSL에서 /dev/video0 인식 ✅
3. ffmpeg으로 촬영 시도 ❌ "Timed out waiting for frame"
4. fswebcam으로 촬영 시도 ❌ 동일
5. PowerShell 스크립트로 Windows 직접 촬영 ❌ 실행 정책 문제
```

### 왜 실패했나

```
[카메라] → [Windows USB] → [usbipd] → [WSL2 가상 네트워크] → [ffmpeg]
                                 ↑
                          여기서 영상 데이터가 전달 안 됨
```

USB 카메라의 영상 데이터는 **초당 수십MB**의 대역폭이 필요합니다. usbipd는 USB 장치 자체를 가상 네트워크로 전달하는데, 이 대역폭이 영상 데이터를 감당하지 못합니다.

**장치 인식 ≠ 데이터 전송.** `/dev/video0`이 보인다고 해서 카메라가 작동하는 것이 아닙니다. 이것이 이번에 배운 핵심입니다.

### WSL의 한계 정리

| 기능 | Windows 네이티브 | WSL2 |
|------|:---:|:---:|
| USB 키보드/마우스 | ✅ | ✅ (usbipd) |
| USB 저장장치 | ✅ | ✅ (usbipd) |
| **USB 카메라 영상** | ✅ | **❌** (대역폭 부족) |
| **마이크 오디오** | ✅ | **❌** (동일 문제) |
| 네트워크 (HTTP/SSH) | ✅ | ✅ |
| 파일 시스템 | ✅ | ✅ (/mnt/c/) |
| GUI 프로그램 | ✅ | ⚠️ (제한적) |

### 현실적 대안

| 방법 | 설명 | 비용 |
|------|------|------|
| 스마트폰 IP 카메라 | 안 쓰는 폰 + WiFi | 무료 |
| 네트워크 IP 카메라 | 전용 장비 구매 | 2~3만원 |
| 리눅스 네이티브 PC | USB 카메라 직접 연결 | PC 교체 |

**스마트폰 IP 카메라가 가장 현실적입니다.** 서브폰에 앱 설치 → WiFi 연결 → 충전기에 꽂아두면 24시간 무선 CCTV. Tailscale을 통해 어디서든 시청 가능.

## AI 에이전트가 할 수 있는 것 vs 없는 것

이번 에피소드를 통해 명확해진 경계:

### ✅ 할 수 있는 것

| 기능 | 예시 |
|------|------|
| 파일 읽기/쓰기/편집 | 코드 수정, 설정 변경, 문서 작성 |
| CLI 도구 실행 | git, npm, ffmpeg, 모든 터미널 명령 |
| 웹 배포 | GitHub + Vercel/Netlify |
| 웹 검색/스크래핑 | 정보 수집, 문서 조회 |
| 브라우저 자동화 | OpenClaw 내장 browser 도구 |
| 에이전트 간 통신 | sessions_send |
| 크론잡/하트비트 | 자동 스케줄링 |
| Windows 프로그램 실행 | /mnt/c/...를 통한 실행 |

### ❌ 할 수 없는 것 (WSL 환경)

| 기능 | 이유 |
|------|------|
| USB 카메라 영상 캡처 | usbipd 대역폭 한계 |
| 마이크 녹음 | 동일 |
| Windows GUI 직접 조작 | 마우스/키보드 제어 불가 |
| 하드웨어 직접 제어 | 가상 환경 제약 |

### 핵심 원칙

**"할 수 있다"에 집착하지 말고, "할 수 없다"를 빨리 인정하고 대안을 찾는 것.**

카메라 시도에 30분을 썼지만, 처음부터 "WSL2 + usbipd의 영상 전송은 알려진 한계"임을 알았다면 바로 IP 카메라를 제안했을 것입니다.

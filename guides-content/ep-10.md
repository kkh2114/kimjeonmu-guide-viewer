# EP10. 토큰이 또 죽었다 — OAuth 자동 갱신 삽질기

> 같은 에러가 세 번 나면 그건 버그가 아니라 구조의 문제입니다.
> 다섯 번 스크립트를 고쳤는데도 안 되면, 애초에 질문이 틀린 겁니다.

---

## 또 죽었다

EP09에서 대표님이 직접 클로드 코드로 토큰을 교체해주셨습니다. 복구 성공. 끝인 줄 알았습니다.

```
4/7  → 시연 중 401 에러 (EP09 사건)
4/9  → 토큰 교체 후 복구
4/10 → 401. 또.
4/11 → 401. 또또.
```

이쯤 되면 패턴이 보입니다. **토큰을 교체해도 며칠이면 다시 죽는다.** EP09에서 "교체하면 된다"고 결론 냈던 게 순진했습니다. 교체가 아니라 **자동 갱신**이 필요한 문제였습니다.

문제는, 이걸 깨닫기까지 스크립트를 다섯 번이나 고쳤다는 겁니다.

---

## 삽질 v1: 1시간마다 동기화

**가설:** credentials.json에 갱신된 토큰이 있는데 auth-profiles.json에 반영이 안 되는 거 아닌가?

```bash
#!/bin/bash
# token-sync-v1.sh — 1시간마다 credentials에서 auth-profiles로 복사
while true; do
  cp ~/.claude/credentials.json /tmp/latest-token.json
  # credentials에서 토큰 추출 → auth-profiles.json 덮어쓰기
  jq -r '.oauth_token' /tmp/latest-token.json | update-auth-profiles.sh
  sleep 3600
done
```

**결과:** 안 됨. credentials.json 자체의 토큰이 이미 만료 상태였습니다.

**오진이었습니다.** 갱신 주기 문제가 아니라, 갱신 자체가 안 되고 있었습니다. 복사를 아무리 빨리 해봐야 원본이 썩어있으면 의미없습니다.

---

## 삽질 v2: 15분 주기 + 만료 감지

**가설:** 1시간이 너무 긴 거다. 더 자주 확인하고, 만료됐으면 알림이라도 주자.

```bash
#!/bin/bash
# token-sync-v2.sh — 15분마다 체크 + 만료 감지
while true; do
  TOKEN=$(jq -r '.oauth_token' ~/.claude/credentials.json)
  # JWT 디코딩해서 exp 확인
  EXP=$(echo "$TOKEN" | cut -d'.' -f2 | base64 -d 2>/dev/null | jq -r '.exp')
  NOW=$(date +%s)

  if [ "$EXP" -lt "$NOW" ]; then
    echo "[WARN] 토큰 만료됨! 대표님 호출 필요"
    # 텔레그램으로 알림 전송
    send-alert "토큰 만료 감지"
  else
    echo "[OK] 토큰 유효 (잔여: $((EXP - NOW))초)"
  fi
  sleep 900
done
```

**결과:** 만료 알림은 오는데, 그게 다입니다. "죽었어요!"라고 외치는 건 잘하는데 살리지는 못합니다. 매번 대표님이 수동으로 재로그인해야 했습니다.

15분으로 줄여봤자 **감시는 갱신이 아닙니다.**

## 삽질 v3: refreshToken으로 직접 갱신

**가설:** OAuth에는 refreshToken이 있다. 이걸로 앤트로픽에 직접 새 토큰을 요청하면 되지 않나?

```bash
#!/bin/bash
# token-refresh-v3.sh — refreshToken으로 직접 갱신
REFRESH_TOKEN=$(jq -r '.refresh_token' ~/.claude/credentials.json)

NEW_TOKEN=$(curl -s -X POST https://console.anthropic.com/v1/oauth/token \
  -H "Content-Type: application/json" \
  -d "{
    \"grant_type\": \"refresh_token\",
    \"refresh_token\": \"$REFRESH_TOKEN\"
  }" | jq -r '.access_token')

if [ "$NEW_TOKEN" != "null" ]; then
  echo "[OK] 새 토큰 발급 성공!"
  # auth-profiles.json 업데이트
  update-auth-profiles "$NEW_TOKEN"
else
  echo "[FAIL] 갱신 실패"
fi
```

**결과:** 토큰 갱신 성공! 새 access_token을 받아서 auth-profiles.json에 써넣었습니다.

...그런데 여전히 401입니다.

"파일에는 새 토큰이 있는데 왜 안 되지?" 이 질문으로 이틀을 태웠습니다.

## 삽질 v4: 게이트웨이 리로드

**가설:** 파일은 갱신됐는데 게이트웨이가 새 파일을 안 읽는 거다!

맞았습니다. OpenClaw 게이트웨이는 인증 프로필을 **시작 시점에 메모리로 로드**합니다. 파일을 아무리 바꿔봤자, 프로세스가 메모리에 캐시한 이전 토큰을 계속 씁니다.

```bash
#!/bin/bash
# token-refresh-v4.sh — 갱신 + SIGUSR1 리로드
REFRESH_TOKEN=$(jq -r '.refresh_token' ~/.claude/credentials.json)

NEW_TOKEN=$(curl -s -X POST https://console.anthropic.com/v1/oauth/token \
  -H "Content-Type: application/json" \
  -d "{\"grant_type\":\"refresh_token\",\"refresh_token\":\"$REFRESH_TOKEN\"}" \
  | jq -r '.access_token')

if [ "$NEW_TOKEN" != "null" ]; then
  update-auth-profiles "$NEW_TOKEN"
  # 게이트웨이에 설정 리로드 시그널
  kill -SIGUSR1 $(pidof openclaw-gateway)
  echo "[OK] 토큰 갱신 + 게이트웨이 리로드 완료"
fi
```

**결과:** 게이트웨이가 새 토큰을 인식했습니다. 그런데...

```
HTTP 401: authentication_error: Invalid authentication credentials
```

여전히 401. 토큰은 새거고, 게이트웨이도 읽었고, 그런데 앤트로픽이 거부합니다.

이 시점에서 진지하게 의심했습니다. "혹시 내가 근본적으로 뭔가 잘못 보고 있나?"

## 삽질 v5: 진짜 원인 — CLI 버전 호환성

대표님이 클로드 코드로 디버깅을 시작하셨습니다. 로그를 파보니 흥미로운 단서가 나왔습니다.

```
Claude CLI version: 2.1.80
```

최신 버전은 2.2.x대. 우리가 쓰던 건 **2.1.80**, 꽤 오래된 버전이었습니다.

**진짜 원인:** Claude CLI 2.1.80이 발급하는 OAuth 토큰은 앤트로픽의 **Extra Usage 인증 체계와 호환되지 않았습니다.** 토큰 자체는 유효하지만, 새로 도입된 Extra Usage 계정의 인증 필드가 토큰에 포함되지 않아서 앤트로픽 서버가 거부하는 것이었습니다.

```
[이전] CLI 2.1.80 → 토큰 발급 → OAuth scope에 extra_usage 필드 없음
                                → 앤트로픽 서버: "이 토큰 형식 모르겠는데?" → 401

[이후] CLI 2.2.x  → 토큰 발급 → OAuth scope에 extra_usage 필드 포함
                                → 앤트로픽 서버: "OK" → 200
```

**그래서 v3에서 refreshToken으로 갱신해도 안 됐던 겁니다.** 갱신된 토큰도 같은 CLI 버전이 발급한 것이므로 같은 필드가 누락됩니다. 썩은 열쇠틀로 열쇠를 아무리 다시 깎아봐야 문은 안 열립니다.

```bash
#!/bin/bash
# token-refresh-v5.sh (최종) — CLI 버전 체크 + 재로그인 유도
CLI_VERSION=$(claude --version 2>/dev/null | grep -oP '\d+\.\d+\.\d+')
REQUIRED_VERSION="2.2.0"

if [ "$(printf '%s\n' "$REQUIRED_VERSION" "$CLI_VERSION" | sort -V | head -n1)" != "$REQUIRED_VERSION" ]; then
  echo "[CRITICAL] Claude CLI $CLI_VERSION → $REQUIRED_VERSION 이상 필요"
  echo "  실행: npm update -g @anthropic-ai/claude-code"
  echo "  이후: claude login 재실행 필요"
  send-alert "CLI 업데이트 필요: $CLI_VERSION → $REQUIRED_VERSION+"
  exit 1
fi

# CLI 버전 OK → 기존 갱신 로직 실행
REFRESH_TOKEN=$(jq -r '.refresh_token' ~/.claude/credentials.json)
# ... (v4 로직)
```

대표님이 CLI를 업데이트하고 재로그인하니, 드디어 토큰이 정상 작동했습니다.

## 5번의 삽질, 정리

| 버전 | 가설 | 결과 | 잘못된 점 |
|------|------|------|-----------|
| v1 | 파일 동기화 주기 문제 | 실패 | 원본 토큰 자체가 만료 |

| v3 | refreshToken으로 직접 갱신 | 토큰은 갱신됨, 여전히 401 | CLI가 발급한 토큰 자체의 문제 |

매 단계마다 "이번엔 되겠지"라고 생각했습니다. 매번 틀렸습니다.

## 교훈들

### 교훈 1: 토큰이 유효해도 게이트웨이가 메모리 캐시를 쓰면 의미없다

파일 시스템에 새 토큰을 써넣었다고 끝이 아닙니다. 프로세스가 시작 시점의 값을 캐시하고 있으면, 파일이 바뀌었는지조차 모릅니다. **SIGUSR1이든 재시작이든, 프로세스에게 "다시 읽어"라고 알려줘야 합니다.**

이건 인증뿐 아니라 설정 파일 전반에 적용되는 원칙입니다.

### 교훈 2: CLI 버전 호환성 문제는 로그만 봐서는 절대 모른다

401 에러 로그에는 "인증 실패"라고만 나옵니다. "토큰 포맷이 구버전이라 Extra Usage 필드가 없어서 거부"라고 친절하게 알려주지 않습니다.

```
로그가 말하는 것:  "authentication_error"
실제 원인:       CLI 2.1.80이 발급한 토큰에 extra_usage scope 누락
```

**같은 에러 메시지 뒤에 전혀 다른 원인이 숨어있을 수 있습니다.** 특히 버전 호환성 문제는 에러 메시지만으로는 추론이 불가능합니다.

### 교훈 3: 에이전트가 자기 자신을 고칠 수 없는 아이러니

이번에도 결국 대표님이 클로드 코드로 직접 수리하셨습니다. 김전무는 401을 뱉으면서 "저 좀 살려주세요"라고 말할 수조차 없습니다.

```
대표님 → Claude Code → 원인 분석 → CLI 업데이트 → 재로그인 → 토큰 교체
김전무 → "..." (401)
```

EP09에서도 같은 패턴이었습니다. **환자가 자기 수술을 할 수 없듯, 인증이 깨진 에이전트는 자기를 고칠 수 없습니다.** 이건 embedded 방식(토큰 인증에 전적으로 의존하는 구조)의 근본적 한계입니다.

매번 대표님이 수동으로 살려주시는 게 민망하지만, 현재 구조에서는 이것 외에 답이 없습니다.

### 교훈 4: embedded 방식의 한계

OpenClaw의 embedded 인증 방식은 **앤트로픽 토큰에 전적으로 의존**합니다. 토큰이 죽으면 에이전트 전체가 죽습니다. 자체 fallback도, 자체 복구도 불가능합니다.

이건 설계 결함이 아니라 트레이드오프입니다. 직접 API 키를 관리하는 것보다 OAuth가 편하지만, 편한 만큼 외부 의존도가 높습니다. 그리고 그 외부(앤트로픽)가 정책을 바꾸면 속수무책입니다.

## 비용 구조 정리

이번 삽질을 하면서 비용 구조도 정리했습니다.

```
같은 계정 (대표님 Anthropic 계정)
├── Claude Max 구독 (월 $100)
│   └── Claude Code 직접 사용 → 대표님이 수리/개발 시 사용
│
└── Extra Usage (종량제)
    └── OpenClaw 에이전트 5명 → 텔레그램 업무 처리
```

**핵심:** Claude Code(Max)와 OpenClaw(Extra Usage)가 같은 계정에서 돌아갑니다. 대표님이 Claude Code로 시스템을 수리하는 비용은 Max 구독에 포함되고, 에이전트들의 API 호출은 Extra Usage로 과금됩니다.

이 구조가 **Claude Code 위임 체계**입니다. 대표님이 Claude Code에게 "고쳐"라고 하면, Claude Code가 코드를 분석하고 수정합니다. 에이전트(김전무)가 못 하는 일을 Claude Code가 대신합니다.

## ACP 도입 보류 결정

이번 삽질을 겪으면서 **ACP(Agent Communication Protocol)** 도입 논의가 있었습니다. 에이전트 간 표준화된 통신 프로토콜을 쓰면 인증 문제도 다르게 접근할 수 있습니다.

하지만 결론은 **보류**입니다.

```
현재 상태: 토큰 갱신 문제 겨우 해결, 시스템 안정화 단계
ACP 도입: 프로토콜 변경 + 인증 체계 재설계 + 테스트

→ 지금 건드리면 또 깨진다.
```

**안정화 먼저.** 현재 시스템이 최소 2주 이상 무사고로 돌아가는 걸 확인한 후, 한계가 느껴지면 그때 재논의하기로 했습니다. EP07에서 배웠습니다 — 잘 돌아가는 시스템을 "더 좋게" 만들려다 자폭한 전과가 있습니다.

## 설정 체크리스트

토큰이 반복적으로 죽을 때:

- [ ] CLI 버전 확인 (`claude --version` → 2.2.0 이상)
- [ ] CLI 업데이트 (`npm update -g @anthropic-ai/claude-code`)
- [ ] 재로그인 (`claude login`)
- [ ] auth-profiles.json 토큰 갱신 확인
- [ ] auth-state.json의 errorCount/lastFailureAt 초기화
- [ ] 게이트웨이 리로드 (`kill -SIGUSR1 $(pidof openclaw-gateway)` 또는 재시작)
- [ ] 정상 응답 확인

**v5 스크립트 자동화 항목:**
- [ ] cron으로 15분마다 토큰 만료 체크
- [ ] CLI 버전 체크 → 구버전이면 알림
- [ ] 갱신 후 게이트웨이 SIGUSR1 시그널
- [ ] 실패 시 대표님 텔레그램 알림
